

## Trust States

| Trust State | Meaning | When it occurs |
|---|---|---|
| `static_verified` | All facts passed static verification only | `verification_mode = "static"` with 0 rejected facts (`src/cluster.ts:131`) |
| `partial_static` | Some facts rejected by static verifier | `verification_mode = "static"` with ≥1 rejected fact (`src/cluster.ts:131`) |
| `llm_verified` | All static-passed facts also passed LLM verification and were promoted | `verification_mode = "llm"` with 0 total rejected facts (`src/cluster.ts:221`) |
| `partial_llm` | Some facts rejected by LLM verifier; promoted facts upgraded to `llm_verified` confidence | `verification_mode = "llm"` with ≥1 total rejected fact (`src/cluster.ts:221`) |
| `unprepared` | No factsheet ever created | Cluster created, never prepared |
| `untrusted` | Factsheet marked stale or needs re-prepare | `markClusterFactsheetStale` / `markClusterNeedsPrepare` (`src/db.ts:867,877`) |

## The distinguishing field: `omitted_facts`

`src/clusterRefresh.ts:176-182` shows the differentiation:

```ts
const omittedFacts = typeof content.omitted_facts === "number" ? content.omitted_facts : 0;
if (status === "llm_verified") {
  return omittedFacts > 0 ? "partial_llm" : "llm_verified";
}
return omittedFacts > 0 ? "partial_static" : "static_verified";
```

`omitted_facts` accumulates rejected counts from both static verifier and LLM verifier (`src/cluster.ts:220`).

## Consult gating

`cluster_consult` (`src/clusterConsult.ts:55`) only trusts a factsheet if:
- `factsheet.status === "llm_verified"`, **or**
- `cluster.static_factsheet_policy === "allow"` (allows `static_verified` too)

So `partial_static` and `partial_llm` are effectively consult-blocked without operator override — they exist as audit artifacts but cannot serve `cluster_consult` calls.
