

When `cluster_consult` detects a stale factsheet (`src/clusterConsult.ts:62-78`):

1. **Mark stale**: calls `db.markClusterFactsheetStale(factsheetId)` and logs `cluster_refresh_required` event
2. **Return error**: `CLUSTER_FACTSHEET_STALE` error with `changed_files` list

The caller-facing MCP layer (`src/tools.ts:781-817`) then decides whether to:
- **Retry** once immediately (handles race where another process refreshed)
- **Strict evidence revalidation** under per-cluster lock — runs `revalidateClusterEvidence()` to re-check every selector/snippet_hash. If it passes, re-calls `consultCluster`. If it fails, falls through.
- **Fallback to V1** `claude_consult` — marks cluster `needs_prepare`, answers via normal session routing, logs `cluster_fallback_to_claude_consult`

The caller **never sees** `CLUSTER_FACTSHEET_STALE` unless `autoRefresh = false` or the error is `CLUSTER_PROJECT_MISMATCH` (a security boundary that must be surfaced).
