Three things happen atomically:

1. **`db.markClusterFactsheetStale()`** — marks the factsheet record as `stale` and sets the cluster's `trust_state` to `untrusted` (`src/db.ts:746-758`).
2. **`cluster_refresh_required` event logged** — records the stale files and factsheet version for observability.
3. **Error returned** — `CLUSTER_FACTSHEET_STALE` error with the list of changed files. The consult does **not** proceed.

The stale check is done by re-hashing every file in `cluster_file_hashes` against the stored hash/size (`src/clusterConsult.ts:199-215`). Only files scoped in the factsheet trigger invalidation — unrelated repo changes are ignored.
