## Route resolution

## Orphan detection and recovery (shared logic, lines 271–282 / 300–311 / 351–363)

```typescript
const exists = await this.runtime.claude.sessionFileExists(session.claude_session_id);
if (!exists) {
    const oldView = this.runtime.db.getSessionView(session.id);
    this.runtime.db.markSessionOrphaned(session.id);   // ← session flagged as orphaned in DB
    return {
        selectedSession: null,                          // ← no file to resume
        selectedView: null,
        bootstrapContext: oldView ? buildBootstrapContext(oldView, "orphaned") : undefined,
        matchScore: 0,
        matchReason: "... was orphaned; creating replacement from registry context.",
        wasOrphanRecovery: true                         // ← logged in event
    };
}
```

**Key behaviors:**
- The session is marked `orphaned` in the DB — it is **not** deleted, so the accumulated session context is preserved.
- A `bootstrapContext` is built from the old session view — this injects the prior conversation summary into the prompt so the new replacement session has context.
- `wasOrphanRecovery: true` is returned, which surfaces in the `routing` field of the response and generates a `new_session` event (not a `consult` event — see line 191).

## Invocation (src/consult.ts:117)

Because `selectedSession` is `null`, `runPrompt()` is called **without** `--resume`, so Claude starts a fresh session. The new session ID returned by Claude is then stored in the DB (lines 155–162).

## Summary

| What happens to the deleted file | Nothing — it's just gone |
|---|---|
| What happens to the DB session record | Marked `orphaned`; context preserved |
| What happens on next consult | New session created; prior context injected via `bootstrapContext`; `wasOrphanRecovery: true` in response |
| What does the user see | A `routing.was_orphan_recovery: true` flag; the session appears as a new session in the DB |

The system never tries to force-reuse a deleted file — it silently pivots to orphan recovery. The old session's history is not lost; it's re-hydrated into the bootstrap prompt for the replacement.
